# torry-pines-shaker-server

REST API for the Torrey Pines Scientific shaker (SC20 series), backed
by the `matterlab_shakers.TorreyPinesShaker` driver from the Matter Lab
[shakers](https://gitlab.com/aspuru-guzik-group/self-driving-lab/devices/shakers)
GitLab repo.

This repo conforms to **lab status spec v1.1** (see
`ac-organic-lab/docs/STATUS_SPEC.md`). Implements the recipe v2 §3.5
`shake` step: the server owns the duration timer, with a watchdog that
stops the motor on process exit.

## Endpoints

Spec endpoints (always available):

- `GET /` &mdash; `ProbeResponse`
- `GET /health` &mdash; `HealthResponse`
- `GET /status` &mdash; full `EquipmentStatus` envelope (side-effect-free)
- `GET /openapi.json` &mdash; generated by FastAPI

Claim protocol (v1.1):

- `POST /control/claim`     &mdash; body `ClaimRequest` -> `ClaimResponse` (HTTP 409 with `ClaimRejection` on conflict)
- `POST /control/heartbeat` &mdash; header `X-Claim-Token`
- `POST /control/release`   &mdash; header `X-Claim-Token` (HTTP 204)

Control (gated by `X-Claim-Token` when `enforce_claims=true`):

| Method + Path | Body | Notes |
|---|---|---|
| `POST /control/startup` | &mdash; | Open the serial port and verify the device. |
| `POST /control/shutdown` | &mdash; | Stop motor, close port. |
| `POST /control/shake/set_temperature` | `{temperature_c: float}` | -20..+110 C. |
| `POST /control/shake/set_speed` | `{speed_level: int}` | 1..9; use `/control/shake/stop` to halt. |
| `POST /control/shake/start` | `{speed_level, temperature_c, duration_s, wait_for_temperature?}` | Server times `duration_s` and stops the motor. HTTP 412 when `wait_for_temperature=true` and the setpoint is not reached within the configured timeout. |
| `POST /control/shake/stop` | &mdash; | Idempotent; cancels watchdog, drops speed to 0. |

`equipment_kind` is `"shaker"` (added to the v1.1 enum in
`ac-organic-lab/docs/STATUS_SPEC.md` alongside this server).

## Recipe v2 §3.5 mapping

A `shake` step from a Recipe v2 plan:

```yaml
- kind: shake
  device: torry_pines_shaker
  plate: A
  speed_level: 5
  temperature_c: 25.0
  duration_s: 3600
  wait_for_temperature: false
```

maps to one `POST /control/shake/start` body:

```json
{
  "speed_level": 5,
  "temperature_c": 25.0,
  "duration_s": 3600,
  "wait_for_temperature": false
}
```

The device server times `duration_s` internally and stops the motor at
the end (or on process exit via the lifespan watchdog). `plate` is
recipe-side metadata and not part of the device API.

## Install

This is a Windows-only deployment in production (the shaker is on a
COM port). Dry-run mode works on macOS / Linux for development and CI.

```powershell
cd C:\Users\sdl2\Projects\torry-pines-shaker-server
uv sync --extra api
copy config.example.toml config.toml   # then edit
```

The vendor driver and its serial dependency are pulled from GitLab via
`tool.uv.sources` in `pyproject.toml`; no PyPI publish needed.

## Run

```powershell
# Dry-run (no hardware required):
uv run --extra api torry-pines-shaker-serve --dry-run

# Real hardware (reads [service] host/port from config.toml):
uv run --extra api torry-pines-shaker-serve
```

To run as a Windows service via NSSM, follow
`ac-organic-lab/docs/DEVICE_PC_SETUP.md` &mdash; the conventional
service name is `torry-pines-shaker` and the port is `8030`.

## Test

```powershell
uv run --extra dev pytest -q
```

All tests use `dry_run=True` so no hardware is required.

## Watchdog contract

Per recipe v2 §3.5, the device server &mdash; not the workflow &mdash;
owns the shake duration timer. Mechanics:

1. `POST /control/shake/start` schedules an `asyncio.Task` that sleeps
   `duration_s` and then stops the motor under the service lock.
2. `POST /control/shake/stop` cancels the task and stops the motor.
3. Service lifespan teardown (Ctrl-C, NSSM stop, crash &mdash; assuming
   it crashes cleanly enough to fire the `lifespan` finalizer) cancels
   the task and stops the motor (`watchdog_stop_on_exit=true` in
   `config.toml`).

A hard kill of the Python process &mdash; e.g. `taskkill /F` or
operator pulls power &mdash; bypasses the lifespan handler. There is
no hardware-level watchdog on the Torrey Pines controller; the motor
will keep spinning at whatever setpoint was last commanded. Use the
NSSM auto-restart wrapper in production so a process crash &mdash;
which still fires `lifespan` &mdash; is the common failure mode.

## Claim enforcement

When `enforce_claims = true` (the default), every `/control/*` request
must carry a valid `X-Claim-Token`. Acquire one with::

    POST /control/claim
    {"owner": "agent:solubility", "session_id": "<uuid>", "ttl_s": 60}

Send heartbeats every `heartbeat_interval_s` (returned by the acquire
response). Release on shutdown. The `lab_skills.ClaimManager` async
context manager does this for you when the device's `equipment.yaml`
entry has `protocol: "1.1"`.

Set `enforce_claims = false` for legacy / single-operator deployments
where the device should still publish `details.claimed_by` and
`allowed_actions` but not reject writes from clients that omit the
header.
